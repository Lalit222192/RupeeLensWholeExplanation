RupeeLens: Complete Project Explanation from Basics to Advanced Implementation

RupeeLens is a Java web-based expense tracking system built with Servlet, JSP, JDBC, MySQL, Bootstrap, and JavaScript. It supports authentication, category management, income and expense tracking, reporting, receipt upload, and an OCR-powered feature that extracts data from receipts and pre-fills the expense form for faster entry.

Implemented: Servlet + JSP + JDBC + MySQL + MVC Implemented: Session, Cookies, CRUD, Reports, File Upload, OCR Not Implemented: JSF, Hibernate Future Scope: password hashing, ORM, advanced analytics
Problem Statement Students and working users often track daily income and expenses manually, which leads to missing entries, poor categorization, weak reporting, and extra time spent typing receipt details. RupeeLens solves this by centralizing financial entries in one web system and reducing manual work through receipt OCR.
Target Users Personal finance users, students, and anyone who wants a simple web-based tracker.
Server Style Request-response Java web app on Tomcat using JSP views and Servlet controllers.
Database MySQL database named RupeeLens with user-specific records.
Special Feature OCR receipt extraction through /expense-ocr with user confirmation before save.

1. Project Overview

This section explains what the project is, why it was built, what it achieves, and why it is relevant in the real world.

Project Name

RupeeLens is an expense and income tracking system that helps users manage money records, categorize spending, upload receipts, and view reports through a web interface.

Core Objective

The main objective is to provide a structured financial tracking system where each logged-in user can create, manage, and analyze income and expense entries securely.

Updated Objectives

  • Allow user registration, login, logout, and session-controlled access.
  • Support separate modules for income, expense, and category management.
  • Generate dashboard analytics and monthly report views.
  • Allow receipt file upload for expense entries.
  • Reduce manual entry effort using OCR-based receipt data extraction.

Real-World Relevance

  • Expense tracking is a real need for students, freelancers, salaried users, and households.
  • Digital receipts are common, but manual form entry is repetitive and error-prone.
  • Reports and charts help users understand where their money is going.
  • OCR improves usability by turning receipt images into structured suggestions.
Domain: Personal Finance Management Pattern: MVC-Oriented Java Web Application

2. Technology Stack

The project uses a standard Java web application stack. Each technology has a specific purpose in the system.

Servlet

Servlets act as controllers. They receive HTTP requests, read parameters, validate user session, call DAOs, and decide whether to forward or redirect.

Why used: Servlets provide strong control over request handling and are ideal for building MVC-style Java web applications.

JSP

JSP pages render forms, tables, dashboard views, and user-facing pages. Data from servlets or DAOs is displayed using JSP tags and scriptlets.

Why used: JSP simplifies server-side page rendering and integrates easily with Servlet-based applications.

JDBC

JDBC connects Java code to MySQL. DAOs use PreparedStatement and ResultSet to run SQL queries and map results to model objects.

Why used: JDBC gives full control over SQL and is appropriate for educational projects focused on database fundamentals.

MySQL

MySQL stores persistent data such as users, categories, income records, and expense records with foreign-key-based relationships.

Why used: It is stable, relational, easy to integrate with JDBC, and supports reporting queries efficiently.

Bootstrap and JavaScript

Bootstrap improves layout, form design, and responsiveness. JavaScript handles receipt preview, search, chart rendering, OCR fetch calls, and client-side events.

Why used: It improves the user experience without changing backend architecture.

Tomcat and Utility Classes

The application runs on Apache Tomcat. Utility classes such as DBConnection and AppContextListener manage connectivity and cleanup.

Why used: They support deployment, JDBC lifecycle management, and runtime stability.

4. Frontend (Client-Side) Explanation

The frontend is implemented using JSP, Bootstrap, CSS, and JavaScript. It focuses on simple form-based interaction, clean dashboard views, and responsive user experience.

UI Structure

  • login.jsp and register.jsp manage authentication UI.
  • dashboard.jsp shows totals, charts, and recent transactions.
  • addIncome.jsp, addExpense.jsp, and addCategory.jsp provide entry forms.
  • viewIncome.jsp, viewExpense.jsp, and viewCategory.jsp display stored records.
  • components/sidebar.jsp and components/navbar.jsp provide reusable layout blocks.

Forms Covered

Login, registration, income form, expense form, category form, report search, and OCR extraction interaction are all visible in the interface.

5. Backend (Server-Side) Explanation

The backend is controlled by Servlets. They manage routing, business flow, session checks, cookies, database operations, file upload, OCR integration, and redirection.

Main Servlets and Their Roles

Servlet URL Purpose
LoginServlet /login Authenticates user, creates session, stores cookies for remembered email and last login.
RegisterServlet /register Registers new user after validating required fields and duplicate email.
LogoutServlet /logout Invalidates session and can clear remembered email cookie.
IncomeServlet /income Handles add, edit, list, and delete flow for income records.
ExpenseServlet /expense Handles add, edit, list, delete, receipt upload, and OCR confirmation checks.
AddCategoryServlet /category Manages user categories with create, update, delete, and list actions.
ExpenseOCRServlet /expense-ocr Accepts uploaded receipt file, calls OCR provider, extracts values, and returns JSON.

Sessions, Cookies, Redirects, and Exceptions

  • Session management: user object is stored in session as loggedUser; timeout is configured for 30 minutes in code and web.xml.
  • Cookie handling: email is stored in cookie rupeelens_remember_email; last login time is stored in rupeelens_last_login; password is not stored in cookies.
  • Authentication and authorization: protected servlets call getSession(false) and reject unauthenticated access by redirecting to login.
  • URL rewriting and redirection: encodeRedirectURL() and encodeURL() are used to preserve session information when needed.
  • Exception handling: form operations are wrapped in try-catch; generic errors route to error.jsp, and missing pages route to error404.jsp.
  • File upload: @MultipartConfig is used in expense and OCR servlets for receipt upload handling.

Backend Flow Highlights

Authentication
The login servlet checks email and password through UserDAO. If successful, a session is created, session timeout is set, cookies are added, and the user is redirected to dashboard.jsp.
Expense Save
The expense servlet reads form values, checks if OCR confirmation is required, saves uploaded receipt file to System.getProperty("user.home") + "/RupeeLensUploads", creates an Expense object, and then calls ExpenseDAO.
OCR Request
The OCR servlet reads the multipart file, checks the user session, calls Google Vision if configured or OCR.Space otherwise, optionally preprocesses the image for retry, extracts amount/date/category/merchant, and returns JSON to the browser.
Cleanup
AppContextListener deregisters JDBC drivers and shuts down MySQL cleanup threads when the web application is destroyed.

ER Relationship Explanation

Users 1 → Many Income One user can have multiple income records.
Users 1 → Many Expenses Each expense belongs to one logged-in user.
Users 1 → Many Categories Users can create their own categories; some default categories can have null user_id.
Categories 1 → Many Expenses Every expense must belong to one category.

Database Notes for Viva

  • Foreign keys enforce data integrity and ownership.
  • DAO queries always filter by user_id for authorization.
  • Dashboard reports use aggregate SQL like SUM, GROUP BY, and date-based filtering.
  • receipt_path stores file name, while physical file is stored in server-side upload directory.

8. OCR Feature (Very Important)

This is the standout feature of RupeeLens. It reduces manual data entry by analyzing a receipt image and returning structured suggestions for amount, date, merchant, category, and description.

What is OCR?

OCR stands for Optical Character Recognition. It converts text inside an image or scanned receipt into machine-readable text.

Why OCR is Used Here

  • Users often have receipt photos but do not want to type everything manually.
  • Expense entry becomes faster and more practical.
  • The system can suggest values, but the user still verifies them before final save.
Important Design Choice: OCR suggests, user confirms, then expense is saved.

Where OCR is Implemented

  • controller.ExpenseOCRServlet for OCR request handling.
  • controller.OcrHealthServlet for OCR configuration check.
  • addExpense.jsp for receipt upload and OCR panel.
  • js/dashboard.js for fetch, preview, response display, and confirmation logic.
  • ExpenseServlet for final save with OCR confirmation check.

OCR Flow in RupeeLens

1
Upload receipt
User selects a receipt image or PDF in addExpense.jsp.
2
Click Extract Receipt Data
JavaScript collects the file using FormData and sends it to /expense-ocr using fetch().
3
OCR servlet processes file
ExpenseOCRServlet reads image bytes, validates session, and chooses the OCR provider.
4
Provider call and retry logic
The servlet tries Google Vision if configured, otherwise OCR.Space. If quality is weak, preprocessing and a second OCR pass may run.
5
Extract structured fields
Regex and parsing logic identify amount, date, merchant, and likely category from OCR text.
6
Auto-fill form
Frontend fills amount, date, category, description, and shows raw text plus confidence information.
7
User confirms
The user checks the confirmation box. Hidden field ocrConfirmed becomes true.
8
Expense save allowed
ExpenseServlet blocks save if OCR was used but not confirmed.

OCR Backend Handling

  • The OCR endpoint is mapped as /expense-ocr and uses @MultipartConfig.
  • Response type is JSON, not HTML, because frontend expects structured data.
  • The servlet returns fields like success, amount, date, categoryName, merchant, description, rawText, confidence, and provider.
  • Fallback suggestions are generated if text is weak or partially unreadable.
  • OCR reliability is improved through timeout handling, second-pass preprocessing, and safe JSON output.

Benefits of the OCR Module

  • Reduces typing effort while adding expenses.
  • Makes the project more innovative than a normal CRUD system.
  • Shows integration of external APIs with Java Servlet architecture.
  • Improves viva value because it combines file upload, JSON response, parsing logic, and user-controlled automation.

9. Security and Best Practices

This section should be explained honestly in viva. It includes what is already implemented and what can be improved further.

Implemented Practices

  • Password is not stored in cookies. Only remembered email and last login timestamp are stored.
  • Protected servlets verify the session before allowing access.
  • DAO methods use PreparedStatement, which helps prevent SQL injection.
  • Expense and category operations are filtered by user_id, which supports authorization at the query level.
  • URL encoding methods are used for redirects.
  • File upload size is limited through @MultipartConfig.
  • OCR does not auto-save financial records without user confirmation.

Important Viva Clarification

  • The current project stores passwords in the database as plain values and checks them directly in UserDAO.
  • This is acceptable for explaining the present implementation, but in production the system should use password hashing such as BCrypt.
  • Cookie values are marked HttpOnly in login logic.
  • Additional improvements for future scope can include CSRF protection, stronger file-type validation, and secure password storage.
Implemented: session check, cookies without password, prepared statements Improvement Suggested: password hashing and stricter upload validation
